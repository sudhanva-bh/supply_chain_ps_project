Object-Relational Mapping (ORM) is a way to bridge object-oriented code and relational databases. Instead of writing SQL for every data operation, the developer writes a mapping that the ORM framework uses to handle persistence automatically. Gilhari, from Software Tree, is an ORM microservice built on the JDX engine \cite{gilhari}. Given compiled Java container classes and a JDX mapping file, it generates a full REST API (GET, POST, PUT, DELETE) for each mapped object with no data access code needed.

The Model Context Protocol (MCP) is an open specification that lets AI agents communicate with external tools and data sources in a standardized way \cite{mcp}. An MCP server registers a set of named, typed tools, each with a description and a JSON Schema for its parameters. A client (such as an agent framework) can list those tools, pick the right one, call it with structured arguments, and receive a structured result. This is how the AI agent in this project queries the database: through safe, schema-validated function calls rather than generated SQL.

The Model Context Protocol (MCP) is an open source specification that standardise the way that AI agents comminicate with external tools (such as other I/O devices) and database systems. An MCP server registers a set of named, typed tools, each with a description and also creates a JSON Schema for its parameters. A client (in our case, the backend of the system) can get a list of the available tools, pick the right one and call it with structured arguements to receive a structured result. This is how the AI agent in this project queries the database, ie. through a safe and schema validated function calls rather than generated SQL, which might be dangerous for data modification tasks.

ORMCP is a Python package from Software Tree that connects Gilhari to the MCP ecosystem \cite{ormcp}. When it starts, it reads the running Gilhari instance's object model and auto-generates an MCP tool for each entity: \texttt{query\_InventoryItems}, \texttt{insert\_StockTransaction}, \texttt{update\_PurchaseOrder}, \texttt{delete\_Supplier}, \texttt{execute\_raw\_sql}, and so on. These are served over standard I/O so any MCP client can discover and call them without writing any integration code.


ORMCP is a software developed by SoftwareTree LLC. that connects an existing Gilhari application to the MCP ecosystem \cite{ormcp}. Upon initialisation, it fetches the object models of the current Gilhari instance and auto-generates an MCP tool for each user defined entity. These, in our case, are \texttt{query\_InventoryItems}, \texttt{insert\_StockTransaction}, \texttt{update\_PurchaseOrder}, \texttt{delete\_Supplier}, \texttt{execute\_raw\_sql}, etc. These are fetched via standard I/O so that any MCP client can discover and call them without writing massive boiler-plate integration code.

LangChain is a Python framework that provides a consistent interface while working with LLMs, tools, and prompt templates \cite{langchain}. It is suitable to function with multiple providers (OpenAI, Google Gemini, Anthropic) through a common API. This functionality is useful when you want to switch models without changing the agent logic. It also helps enforce a response schema in agentic responses. In this project it wraps the ORMCP tools as \texttt{StructuredTool} objects and enforces the response schema using \texttt{with\_structured\_output()}.

LangGraph sits on top of LangChain and handles stateful, multi-step agent loops \cite{langgraph}. It models the agent as a directed graph where each node does one thing (invoke the LLM, or run a tool call) and edges define what happens next. The full message history is kept in persistent state. This way the agent can accumulate data over several tool calls before producing a final answer. The graph helps the AI agent to choose various paths and backtrack before coming to a final conclusive answer, which reduces token consumption as the need for multiple redundant calls would be eliminated.

Generative UI is a pattern where the AI agent returns a structured JSON payload, instead of free text, and the frontend renders an appropriate widget based on a type specified in the payload \cite{genui}. Rather than just describing data in words, the agent selects the right visualization that has been pre defined (chart, Kanban board, KPI cards, form) and fills it with the actual query results based on the user's natural language query. Flutter appropriately converts this into a dart class using \texttt{.fromJson()} methods from the JSON payload. In this project the agent's output is validated against a Pydantic \texttt{AgentResponse} model with eleven possible response types. This is further enforced by LangChain's \texttt{with\_structured\_output()} API.

The setup uses Docker Compose with two services. The \texttt{sqlserver} service runs the official SQL Server 2022 image on port \texttt{1433} with data persisted in a volume (\texttt{sqlserver\_data}). The \texttt{gilhari-service} is a custom image that extends \texttt{softwaretree/gilhari}. The \texttt{Dockerfile} downloads the Microsoft JDBC driver at build time using \texttt{wget}. Then it adds the compiled \texttt{bin/} classes, the \texttt{config/} directory, and \texttt{gilhari\_service.config}. After all this, Gilhari starts on internal port \texttt{8081}, mapped to port \texttt{80} on the host.

The ORMCP server runs as a child process of the FastAPI app. It is managed through FastAPI's \texttt{lifespan} context and on startup it connects to Gilhari, reads the object model, and registers MCP tools for all six entities. Then the tool set covers query, insert, update, and delete for each entity, and \texttt{execute\_raw\_sql} for joins and aggregations, and \texttt{getObjectModelSummary}. Connectivity was tested using \texttt{test\_mcp\_client.py}, which lists all tool schemas. Logs are stored in \texttt{mcp\_tools\_output.log}.

The agent is in \texttt{backend/agent.py}, implemented as a LangGraph \texttt{StateGraph}. State holds the message history as a list of LangChain message objects (\texttt{HumanMessage}, \texttt{AIMessage}, \texttt{ToolMessage}). The graph has two nodes:

\textbf{Node 1: \texttt{call\_model}} calls the LLM with the current history and the full list of ORMCP tools wrapped as LangChain \texttt{StructuredTool} objects. The LLM provider is set by the \texttt{LLM\_PROVIDER} environment variable: \texttt{google} loads \texttt{ChatGoogleGenerativeAI} (Gemini Flash by default), and \texttt{openai} loads \texttt{ChatOpenAI} (GPT-4o-mini by default). Since both use LangChain's common interface, the agent logic is identical regardless of which provider is active. If the model returns tool calls, execution moves to \texttt{execute\_tools}. If it returns a plain message, the loop ends.

---


\textbf{Node 2: \texttt{execute\_tools}} runs each tool call via \texttt{MCPClient.call\_tool()}. Then it appends the results as \texttt{ToolMessage} objects, and sends the execution back to \texttt{call\_model}.

The system prompt tells the agent to use \texttt{getObjectModelSummary} first on complex queries. This is to help the AI Model get context on how the database is structured and to prevent hallucinations. Furthermore, it instructs to prefer the named ORMCP tools for simple reads and \texttt{execute\_raw\_sql} for joins, and not to use raw SQL for any write operations, which is blocked from execution anyway through direct SQL statements.

The LLM is called once more using \texttt{with\_structured\_output(AgentResponse)} to fetch the final Generative UI payload. Then using NDJSON, this is streamed to the Flutter Admin Dashboard to provide realtime Generative UI to the admin.

\begin{itemize}
    \item \textbf{Get the JDX mapping right first.} My first mistake was in the JDX mapping where a FK reference was ignored. This was overlooked and found out during testing, and I had to rebuild my seed for it to work properly. Also, A mismatch between the JDX class names, \texttt{classnames\_map.json}, and the actual Java classes fails silently and is a pain to debug later, so it's better to check that before writing anything else.

    \item \textbf{Use Pydantic schemas for the Generative UI.} Without using it, LLM outputs can sometimes be unpredictable. A simple "Here if your generated UI without any header text:" can ruin the entire widget, and cause it to crash, preventing which without a proper schema is near impossible. Defining schemas upfront and using LangChain's \texttt{with\_structured\_output()} keeps the format consistent, otherwise, as stated earlier, the frontend renderer can break unpredictably.
    
    \item \textbf{Set up multi-provider support early.} OpenAI tokens are relatively more expensive. It is easy to burn through 20 dollars worth of tokens during unoptimised testing, just if one agentic loop runs forever. LangChain's model abstractions make it easy to swap LLM providers without touching agent code. This can also be used to benchmark the model's quality and accuracy.

    \item \textbf{Check Azure's platform-specific docs before deploying.} Sometimes, the tech stack that you use may not be natively supported for direct deployment. The same for with my Flutter Web App. Hence, it is advicable to check them before hand so that you can setup custom scripts or GitHub Actions workflow for a smooth deployment experience.
\end{itemize}
