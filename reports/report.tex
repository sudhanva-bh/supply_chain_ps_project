Practice School I at Software Tree was structured in two phases. The first phase served as a foundational exercise: building an app-specific Gilhari Library Management system from scratch. That project was completed as a guided practice to develop core proficiency with the Gilhari and JDX ORM toolchain \cite{gilhari}.

With the foundational skills firmly established, the internship transitioned to a significantly more advanced undertaking --- the primary deliverable of PS-I. The assigned project (Project 23) required designing and implementing a production-grade Agentic Supply Chain Intelligence Platform integrating cloud infrastructure, Large Language Models (LLMs), the Model Context Protocol, and a Generative UI system. This report covers this second, primary phase of the internship.

The project required the development of an AI agent using the OpenAI Agents SDK and the LangChain/LangGraph framework to interact with a relational database hosted on Microsoft Azure, through ORMCP --- a semantic gateway developed by Software Tree. The formal problem statement encompassed the following tasks:

\begin{itemize}[itemsep=0pt]
\item \textbf{Cloud Deployment}: Configure, build, and run a Dockerized Gilhari microservice on Microsoft Azure using a free-tier developer license.
\item \textbf{Database Management}: Set up and interact with a Microsoft Azure SQL Server database using an enterprise-grade, six-table normalized schema representing a complete supply chain system.
\item \textbf{AI Agent Development}: Build an intelligent AI agent using OpenAI's tools and/or LangChain/LangGraph that interacts with the relational database through ORMCP's typed MCP tool interface, translating natural language queries into structured database operations.
\item \textbf{Frontend Integration}: Develop a web admin dashboard supporting both standard manual CRUD operations and an agentic AI chat panel rendering dynamic, data-driven UI components.
\item \textbf{Version Control and Documentation}: Maintain the codebase on GitHub with a comprehensive README file and complete project documentation.
\end{itemize}

\section{Object-Relational Mapping and the Gilhari Framework}

Object-Relational Mapping (ORM) is a programming technique that creates a correspondence between object-oriented data representations and their persistence in relational database tables. Rather than writing explicit SQL for every data access operation, the developer defines a declarative mapping that the ORM framework uses to handle all translation automatically. Gilhari, developed by Software Tree, is an app-specific ORM microservice built on the JDX ORM engine \cite{gilhari}. Given a set of compiled Java container classes and a JDX mapping specification, Gilhari automatically generates a complete RESTful API surface (GET, POST, PUT, DELETE) for each mapped object, eliminating the need for any hand-written data access code.

\section{JDX ORM Specification Language}

The JDX grammar file (\texttt{.jdx}) is the declarative configuration artifact that drives Gilhari \cite{jdx}. It specifies the database type and JDBC connection parameters, the Java object model package, and for each class: the corresponding database table name, all virtual attributes with their Java types, primary key designations, and inter-object relationships. JDX supports BYVALUE containment relationships, in which parent objects own their child collections, enabling nested JSON object representations. In the supply chain project, for example, a \texttt{PurchaseOrder} contains a BYVALUE array of \texttt{PurchaseOrderItem} objects, and a \texttt{Supplier} contains its associated \texttt{InventoryItem} records.

\section{Model Context Protocol}

The Model Context Protocol (MCP) is an open standard specification for providing a standardized interface through which AI agents and Large Language Models can communicate with external data sources and tools \cite{mcp}. MCP defines a JSON-RPC-based protocol for tool discovery and tool invocation. An MCP server exposes a catalogue of typed tools, each described by a name, a human-readable description, and a JSON Schema-validated parameter list. An MCP client (such as an agent's orchestration layer) can list available tools, select the appropriate one for a given query, and invoke it with structured arguments, receiving a structured result. This design allows AI models to interact with databases and APIs through safe, schema-validated function calls rather than generating free-form SQL.

\section{ORMCP: The Semantic Gateway}

ORMCP is a Python package developed by Software Tree that serves as a semantic bridge between the Gilhari REST microservice and any MCP-compatible AI agent \cite{ormcp}. On startup, the ORMCP server introspects the running Gilhari instance by reading its object model configuration. It then automatically generates named, typed MCP tools for each entity in the model --- for example, \texttt{query\_InventoryItems}, \texttt{insert\_StockTransaction}, \texttt{update\_PurchaseOrder}, \texttt{delete\_Supplier}, and \texttt{execute\_raw\_sql}. These tools are registered as an MCP server accessible over standard I/O, allowing any MCP-capable client to discover and invoke them without any custom integration code.

\section{LangChain and LangGraph}

LangChain is an open-source Python framework providing standardized abstractions for building LLM-powered applications \cite{langchain}. It offers unified interfaces for connecting to multiple LLM providers, managing prompt templates, and wrapping external tools in a format callable by the LLM. In this project, LangChain is used to wrap ORMCP MCP tools as \texttt{StructuredTool} objects and to enforce the \texttt{AgentResponse} Pydantic schema via \texttt{with\_structured\_output()}.

LangGraph is a stateful graph-based orchestration framework built on top of LangChain \cite{langgraph}. It models an agent's decision process as a directed graph of nodes and edges. Each node represents a discrete step, and edges define transitions based on the agent's output. LangGraph manages the full message history as persistent state, enabling iterative multi-turn tool-calling loops in which the agent accumulates data across multiple database queries before formulating its final response.

\section{FastAPI}

FastAPI is a modern, high-performance Python web framework for building APIs, built on Starlette and Pydantic \cite{fastapi}. It supports asynchronous request handling natively and provides automatic OpenAPI documentation generation. In this project, FastAPI serves as the HTTP entry point for the agentic backend, exposing endpoints for the AI chat interface, direct tool execution, authentication verification, and a reverse proxy to the Gilhari REST service. FastAPI's native \texttt{StreamingResponse} is used to deliver intermediate agent status updates and the final structured response to the Flutter frontend as Newline-Delimited JSON (NDJSON).

\section{Microsoft SQL Server and Azure SQL}

Microsoft SQL Server is an enterprise-grade relational database management system (RDBMS) \cite{mssql}. This project uses the 2022 Developer edition locally via Docker and Azure SQL Database for cloud deployment. Azure SQL Database is a fully managed cloud service on Microsoft Azure providing the SQL Server engine without manual server administration. The Gilhari microservice communicates with SQL Server using the Microsoft JDBC Driver for SQL Server (\texttt{mssql-jdbc-12.4.2.jre8.jar}), which is downloaded at Docker image build time.

\section{Microsoft Azure Cloud Infrastructure}

Microsoft Azure is a cloud computing platform offering a broad range of services \cite{azure}. This project utilises four Azure services:
\begin{itemize}[itemsep=0pt]
\item \textbf{Azure SQL Database}: Managed cloud relational database hosting the six-table supply chain schema (free tier: 100,000 vCore-seconds and 32 GB/month).
\item \textbf{Azure Container Apps}: Serverless container hosting used to run the Dockerized Gilhari microservice (free tier: 180,000 vCPU-seconds/month).
\item \textbf{Azure App Service}: Managed platform-as-a-service environment hosting the Python FastAPI agentic backend (Free F1 tier).
\item \textbf{Azure Static Web Apps}: Hosting service for compiled Flutter web assets, connected to the GitHub repository for automated continuous deployment (Free tier).
\end{itemize}

\section{Flutter Web and Riverpod}

Flutter is a cross-platform UI framework developed by Google that compiles to native code or web JavaScript \cite{flutter}. The admin dashboard is a Flutter web application. It uses \textbf{Riverpod} for reactive state management, \textbf{fl\_chart} for bar, pie, and line chart rendering, \textbf{flutter\_markdown} for rendering LLM text responses, and \textbf{flutter\_svg} for the regional world-map visualization widget. The design uses a dark-mode-first approach with Google Fonts typography and smooth animated state transitions.

\section{Generative UI}

Generative UI is a paradigm in which an AI agent's response is not free-form text but a structured, typed payload that the client application renders as an appropriate interactive interface component \cite{genui}. Rather than returning a textual description of data, the agent selects the most suitable visualization for the data it retrieved (a chart, a Kanban board, a KPI summary, a data table) and populates it with the actual queried values. In this project, the agent's response conforms to a Pydantic-validated \texttt{AgentResponse} schema with eleven distinct response types, enforced using LangChain's \texttt{with\_structured\_output()} API.

\section{JDX ORM Mapping and Java Container Classes}

Six Java container classes were created under the \texttt{com.supplychain.model} package, one per database entity. Each class extends the \texttt{JDX\_JSONObject} base class provided by the JDX ORM framework. The \texttt{config/supply\_chain.jdx} file defines the complete ORM mapping, specifying the Microsoft SQL Server JDBC connection string, the object model package, and for each class: the table name, all virtual attributes with their Java types, the primary key designation, and BYVALUE relationships. A representative excerpt is shown below:

\begin{verbatim}
JDX_DATABASE JDX:jdbc:sqlserver://sqlserver:1433;
   databaseName=supply_chain_db;encrypt=true;
   trustServerCertificate=true;USER=sa;...
   JDX_DBTYPE=SQLSERVER

CLASS .Supplier TABLE Suppliers
    VIRTUAL_ATTRIB supplierID ATTRIB_TYPE int
    VIRTUAL_ATTRIB companyName ATTRIB_TYPE java.lang.String
    VIRTUAL_ATTRIB contactEmail ATTRIB_TYPE java.lang.String
    VIRTUAL_ATTRIB region ATTRIB_TYPE java.lang.String
    PRIMARY_KEY supplierID
    RELATIONSHIP inventoryItems
        REFERENCES ArrayInventoryItem BYVALUE WITH supplierID
;

CLASS .PurchaseOrder TABLE PurchaseOrders
    VIRTUAL_ATTRIB orderID ATTRIB_TYPE int
    VIRTUAL_ATTRIB orderDate ATTRIB_TYPE java.lang.String
    VIRTUAL_ATTRIB totalCost ATTRIB_TYPE double
    VIRTUAL_ATTRIB deliveryStatus ATTRIB_TYPE java.lang.String
    PRIMARY_KEY orderID
    RELATIONSHIP purchaseOrderItems
        REFERENCES ArrayPurchaseOrderItem BYVALUE WITH orderID
;
\end{verbatim}

The Java source files were compiled using \texttt{compile.cmd} into the \texttt{bin/} directory. The \texttt{config/classnames\_map.json} file provides Gilhari with the mapping between simple class names and fully qualified Java names, which is required for correct REST endpoint resolution.

\section{Docker Infrastructure}

The local development infrastructure is defined in \texttt{docker-compose.yml} and consists of two services. The \texttt{sqlserver} service runs the official Microsoft SQL Server 2022 Docker image on port \texttt{1433} with data persisted to a named Docker volume (\texttt{sqlserver\_data}). The \texttt{gilhari-service} builds a custom image extending \texttt{softwaretree/gilhari}; the \texttt{Dockerfile} installs the Microsoft JDBC driver (\texttt{mssql-jdbc-12.4.2.jre8.jar}) via \texttt{wget} at build time, then packages the compiled \texttt{bin/} classes, \texttt{config/} JDX mapping, and \texttt{gilhari\_service.config}. The Gilhari REST server starts on port \texttt{8081} inside the container, exposed as port \texttt{80} on the host. The entire infrastructure is launched with a single command:
\begin{verbatim}
docker-compose up -d --build
\end{verbatim}

\section{ORMCP Semantic Gateway Setup}

The ORMCP server is a Python process spawned as a child of the FastAPI backend on startup, managed through the FastAPI \texttt{lifespan} context manager. On initialization, ORMCP connects to the running Gilhari instance at \texttt{http://localhost:80/gilhari/v1/}, introspects the object model summary, and registers typed MCP tools for all six entities. The resulting tool catalogue includes operations for each entity (query, insert, update, delete) as well as a \texttt{execute\_raw\_sql} utility for JOINs and aggregations, and a \texttt{getObjectModelSummary} introspection tool. The \texttt{test\_mcp\_client.py} script performs a handshake with the ORMCP server, lists all tool schemas, and writes the full output to \texttt{mcp\_tools\_output.log}.

\section{LangGraph Agentic Backend}

The agentic logic is implemented in \texttt{backend/agent.py} as a LangGraph \texttt{StateGraph}. The graph state holds the full message history as a list of LangChain message objects (\texttt{HumanMessage}, \texttt{AIMessage}, \texttt{ToolMessage}). Two nodes implement the core loop:

\textbf{Node 1: \texttt{call\_model}} invokes the configured LLM with the current message history and the complete set of ORMCP tools wrapped as LangChain \texttt{StructuredTool} objects. The LLM provider is selected via the \texttt{LLM\_PROVIDER} environment variable: \texttt{google} loads \texttt{ChatGoogleGenerativeAI} (Gemini Flash by default), and \texttt{openai} loads \texttt{ChatOpenAI} (GPT-4o-mini by default). Both providers are unified through LangChain's model interface, so the agent and tool-calling logic require no provider-specific changes. If the model returns tool calls, the state transitions to \texttt{execute\_tools}. If it returns a plain message with no tool calls, the loop terminates.

\textbf{Node 2: \texttt{execute\_tools}} iterates over all tool calls in the last model message and executes each via \texttt{MCPClient.call\_tool()}. Results are appended as \texttt{ToolMessage} objects. The state transitions back to \texttt{call\_model}.

\textbf{Final Step: Structured Output.} After the tool-calling loop exits, the LLM is invoked one final time using \texttt{with\_structured\_output(AgentResponse)}, forcing its output to conform to the Pydantic response schema. This produces the Generative UI payload streamed to the frontend as NDJSON.

The system prompt instructs the agent to: always call \texttt{getObjectModelSummary} first on complex multi-table queries; prefer dedicated ORMCP tools for single-table reads and \texttt{execute\_raw\_sql} for JOINs; never use raw SQL for mutations; and stop calling tools once sufficient data has been gathered.

\section{Flutter Admin Dashboard}

The Flutter web dashboard is a single-page application with a persistent left sidebar for navigation. The application uses Riverpod for reactive state management and is organized into two functional areas:

\textbf{Manual CRUD Views (Pathway A):} Six data grid views, one per entity, each implemented using a reusable \texttt{AsyncDataGrid} widget providing paginated browsing. A toolbar in each view supports add, edit, and delete actions. Record creation and editing are handled through per-entity form widgets presented in a \texttt{BaseModal} dialog. The \texttt{PurchaseOrderForm} is the most complex, managing nested \texttt{PurchaseOrderItems} in a coordinated sequence of API calls.

\textbf{AI Analytics View (Pathway B):} The \texttt{agent\_chat\_view.dart} component maintains a scrollable conversation history. User messages trigger streaming NDJSON requests to \texttt{POST /api/agentic-chat}. Intermediate status lines (e.g., ``Calling tool: query\_InventoryItems...'') are rendered as progress indicators. The final NDJSON line is parsed as an \texttt{AgentResponse} and passed to \texttt{DynamicMessageWidget}, which dispatches rendering to the appropriate Generative UI sub-widget based on \texttt{response\_type}.

\section{Cloud Deployment on Azure}

The complete system was deployed to Microsoft Azure in five sequential steps following the project's \texttt{cloud\_deployment.md} guide:

\begin{enumerate}[itemsep=0pt]
\item \textbf{Azure SQL Database}: Provisioned on the free tier. The six-table schema was initialized by running \texttt{sql/init.sql} against the Azure SQL endpoint. The JDX configuration (\texttt{supply\_chain.jdx}) was updated with the Azure server hostname.

\item \textbf{Gilhari on Azure Container Apps}: The Gilhari Docker image was built locally and pushed to an Azure Container Registry (ACR). A Container App was created from the ACR image with HTTP ingress on port \texttt{8081}, yielding a public HTTPS URL for the Gilhari REST API.

\item \textbf{FastAPI Backend on Azure App Service}: The \texttt{backend/} directory was deployed as a Python 3.11 web application on an Azure App Service Free F1 tier via the Azure CLI (\texttt{az webapp up}). The \texttt{GILHARI\_BASE\_URL} environment variable was set to the Azure Container App URL, and the LLM API keys were configured as App Service environment variables.

\item \textbf{Flutter Dashboard on Azure Static Web Apps}: The Flutter app was compiled (\texttt{flutter build web --release}) and deployed to Azure Static Web Apps (Free tier) connected to the GitHub repository for CI/CD. The frontend API base URL was updated to point to the deployed App Service.

\item \textbf{End-to-End Verification}: The live Flutter dashboard was accessed, authenticated with the application password, and used to confirm that natural language queries successfully retrieved data from the Azure SQL Database via the deployed Gilhari ORMCP service.
\end{enumerate}

\section{Discussion}

A key technical insight gained from this project is the layered decoupling that the Gilhari-ORMCP-LangGraph stack provides. The developer writes no SQL in the AI orchestration layer; the LLM interacts exclusively with safe, schema-validated MCP tool functions. This eliminates a significant class of potential security and correctness issues (such as SQL injection through LLM-generated queries) and makes the system highly maintainable as the database schema evolves.

The dual-provider LLM architecture, unified through LangChain's model abstraction, proved highly practical. Switching between OpenAI GPT-4o and Google Gemini Flash required only changing the \texttt{LLM\_PROVIDER} environment variable, with zero changes to the agent graph or tool-calling logic. This makes the system resilient to API rate limits and pricing changes.

The Generative UI paradigm proved significantly more valuable than a plain text-based chatbot interface. When the agent returned a Kanban board or a KPI card, the data was immediately comprehensible and actionable in a way that a text paragraph could not match. This highlighted the importance of structured output enforcement via Pydantic and \texttt{with\_structured\_output()} for LLM applications in operational environments.

One area requiring careful attention was the strict alignment required between JDX class names, the \texttt{classnames\_map.json} file, the compiled \texttt{.class} files, and the ORMCP tool names. A naming inconsistency in any of these layers causes silent failures that are difficult to diagnose. Systematic naming conventions and the \texttt{test\_mcp\_client.py} validation script are essential mitigations.

\section{Conclusion}

This project successfully delivered a production-grade, five-layer Agentic Supply Chain Intelligence Platform. Starting from the foundational Gilhari ORM skills acquired through the initial Library Management practice project, the work advanced to address enterprise-scale challenges: a normalized six-table Azure SQL Server schema, cloud deployment across four Microsoft Azure services, an ORMCP semantic gateway, a LangGraph agentic tool-calling loop supporting both OpenAI and Google Gemini models, and a Flutter web dashboard with eleven distinct Generative UI response types.

The Gilhari-ORMCP-LangGraph architecture demonstrated viability as a pattern for building AI-native data applications. By combining Gilhari's zero-boilerplate REST generation with ORMCP's MCP tool auto-discovery, the integration code required to connect an LLM to a relational database was reduced dramatically. LangGraph provided the necessary structure for reliable, multi-turn, multi-tool agentic reasoning, and the Generative UI system transformed the agent's output from plain text into actionable, rendered interfaces that are immediately useful to a supply chain administrator.

\section{Future Work}

Several directions exist for extending and improving this system:
\begin{itemize}[itemsep=0pt]
\item \textbf{Real-time Streaming}: Integrating WebSocket-based streaming between the FastAPI backend and the Flutter frontend would reduce perceived latency for complex, multi-tool agent queries.
\item \textbf{Multi-Agent Orchestration}: Introducing specialized sub-agents (e.g., a dedicated procurement agent and an inventory analyst agent) coordinated by a supervisor would improve performance on complex, multi-domain queries.
\item \textbf{Expanded Database Support}: Testing the ORMCP-Gilhari stack against additional relational databases (PostgreSQL, MySQL) would validate the portability of the architecture.
\item \textbf{Enterprise Authentication}: Replacing the simple shared application password with OAuth 2.0 and role-based access control would make the platform suitable for real enterprise deployments.
\item \textbf{Agent Memory}: Adding a persistent vector-based conversation memory store would allow the agent to maintain context across user sessions and provide more relevant responses over time.
\end{itemize}

\chapter{Recommendations}
% =========================================================

Based on the completed work and the challenges encountered, the following recommendations are offered for developers working with the Gilhari-ORMCP-LangGraph stack:

\begin{itemize}
\item \textbf{Start with the JDX Mapping}: The JDX file is the single source of truth for the entire system. Invest time upfront in getting the naming conventions consistent --- between the JDX class names, the \texttt{classnames\_map.json}, and the Java source files --- before writing any other layer. Naming errors propagate silently through every layer above.

\item \textbf{Validate ORMCP Early}: Run the MCP handshake test (\texttt{test\_mcp\_client.py}) as the very first integration test after starting the Gilhari container. Confirming that the full tool catalogue is exposed before writing the agent loop saves significant debugging time.

\item \textbf{Use Structured Outputs for Generative UI}: Define Pydantic schemas for all response types from the beginning, rather than attempting to parse free-form LLM text. LangChain's \texttt{with\_structured\_output()} API enforces these schemas reliably and is essential for a robust Generative UI system.

\item \textbf{Design for LLM Provider Flexibility}: Use LangChain's model abstractions from the start and avoid any provider-specific API calls in the core agent logic. This allows switching between OpenAI and Gemini via a single environment variable change, which is invaluable for cost management and rate limit mitigation during development.

\item \textbf{Document the Object Model Thoroughly}: The accuracy of the ORMCP system prompt depends on the LLM correctly understanding the object model to select the right tools and field names. A well-structured \texttt{getObjectModelSummary} response is critical for agent accuracy on complex multi-table queries.
\end{itemize}
